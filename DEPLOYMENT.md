# 🚀 Deployment Guide - Meet Cute

## Automatic Cache Busting

Meet Cute uses **automatic cache busting** for Railway deployments. Every git commit triggers a fresh Docker build.

### How It Works

```dockerfile
# Railway provides RAILWAY_GIT_COMMIT_SHA automatically
ARG RAILWAY_GIT_COMMIT_SHA=local
ARG CACHEBUST=${RAILWAY_GIT_COMMIT_SHA}
```

**Benefits:**
- ✅ **No manual CACHEBUST updates** needed
- ✅ **Every commit = fresh build** (no stale cache)
- ✅ **Automatic** - works out of the box on Railway
- ✅ **Traceable** - logs show exact commit SHA being built

### Deployment Flow

1. **Make changes** to code
2. **Commit** to git
3. **Push** to GitHub
4. **Railway detects** new commit
5. **Docker builds** with commit SHA as CACHEBUST
6. **Fresh build** every time - no cache issues

## Deployment Checklist

### Before Deploying

- [ ] All tests pass locally

- [ ] Environment variables are set in Railway

## Troubleshooting

### Build Fails with TypeScript Errors

**Cause:** Code has type errors  
**Solution:** Run `npm run build` locally, fix errors, commit, push

### Build Succeeds but App Crashes

**Cause:** Missing environment variables or database migration issues  
**Solution:** 
1. Check Railway logs for error messages
2. Verify all required env vars are set
3. Check database connection
4. Review `start.sh` migration logs

### Stale Cache (Shouldn't Happen Anymore!)

**Cause:** Old manual CACHEBUST system  
**Solution:** Now automatic! Every commit gets a fresh build using the commit SHA.

### Database Migration Fails

**Cause:** Schema conflicts or connection issues  
**Solution:**
1. Check `DATABASE_URL` is correct
2. Review migration files in `prisma/migrations/`
3. Manually run: `npx prisma migrate deploy`
4. Check Railway database logs

## Manual Deployment (Local Testing)

### Build Docker Image Locally

```bash
# Build with local tag
docker build -t meet-cute:local .

# Run locally
docker run -p 3000:3000 \
  -e DATABASE_URL="postgresql://..." \
  -e OPENAI_API_KEY="sk-..." \
  meet-cute:local
```

### Test Production Build

```bash
# Backend
npm run build

# Frontend
cd src/frontend && npm run build

# Start production server
npm start
```

## Performance Optimization

### Build Time Optimization

Current build uses layer caching:
- ✅ OS packages cached (rarely change)
- ✅ npm dependencies cached (only rebuild when package.json changes)
- ✅ Application code rebuilt every commit (automatic CACHEBUST)

### Runtime Optimization

- Uses Node.js 18 slim image (smaller size)
- Prisma Client pre-generated during build
- Frontend pre-built and served as static files
- Database migrations run once on startup

## Security Notes

### Secrets Management

- ✅ All secrets stored in Railway environment variables
- ✅ Never commit secrets to git
- ✅ Use `.env.example` for documentation only
- ✅ Rotate secrets regularly (especially API keys)

### Database Security

- ✅ DATABASE_URL includes credentials
- ✅ SSL/TLS enabled by default
- ✅ Prisma uses prepared statements (SQL injection protection)
- ✅ Railway database is private (not publicly accessible)
